## 2단계: 연쇄 영향 검증

### 2.1 LangGraph 2단계 재설계

**1단계 (Phase A, 1~9주)**  
`parse_datasheet → generate_code → verify_simulation → self_correct` (FAIL → generate_code, **최대 5회**)

- `verify_simulation`: T-802 `capture_screenshot()` → T-603/604
- T-602 원근보정: 시뮬 경로 **bypass** (1024×600 정면 PNG)
- 게이트: [하] 샘플 **3회 연속 PASS**, 위젯 오차 **±5%**

**2단계 (Phase HW, 10~12주, `--mode hw`)**  
sim_gate_passed 후: `build_and_flash → verify_physical → self_correct_hw` (**최대 2회**)

### 2.2 마일스톤 재정의

| 마일스톤 | 시점 | 게이트 |
|---|---|---|
| M1 | 7주차 | sim [하]/[중] 3회 이내 PASS, report.md |
| M2 | 9주차 | `p10 run --mode sim` 완료, metrics_sim.json |
| M3 | 12주차 | hw HIL, 실기 기동률 70%+, Vision 75%+ |

### 2.3 주차별 공수 재검산

- 전체 task 카드 공수: 55.5인일. 담당별로는 개발자 A 25.5인일, 개발자 B 26.0인일, A+B 페어 4.0인일(T-012 2.0 + T-903 2.0)이다. Phase별로는 Phase A 32.5인일, Phase B 선택 5.5인일, Phase HW 기준 실행 15.5인일, CoreS3 컷 보류 2.0인일로 재분류한다. 컷 보류를 제외한 **실제 실행 공수는 53.5인일**이다. (상세 집계는 `단위구현계획서.md` 제1.8장 공수 재집계 검증표 참조)
- Phase HW(10~12주) 기준 실행 공수 15.5인일 중 B 직접 작업은 12인일 기준선을 약간 초과한다. 따라서 T-006/T-504는 기본 컷으로 유지하고, 12주차 T-903은 A+B 페어로 전환하며 T-905는 예비 버퍼 1일을 사용해 압축한다.
- 트레이드오프: T-006/T-504 기본 컷, T-603/604 시뮬 선완료, M2=Phase HW 착수 조건, 12주차 T-903 페어화·버퍼 사용을 명시한다. 이 조건이 깨지면 Phase B(T-850~T-852)를 즉시 컷한다.
