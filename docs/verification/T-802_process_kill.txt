1. 구현 요약
--------------------------------------------------------------------
src/simulator/sim_driver.py — SimDriver 클래스
  - build(): T-801 관례(build_sim/bin/lvgl_simulator(.exe))대로 cmake configure +
    cmake --build를 subprocess로 구동. 빌드 시작 전 kill_leftover_processes()를
    선제 호출한다(카드 12항).
  - kill_leftover_processes(): Windows에서 `taskkill /F /IM lvgl_simulator.exe`
    (Linux 대비 `pkill -f lvgl_simulator`)를 실행해 이전 실행이 남긴 고스트
    프로세스를 강제 종료한다. 대상이 없어도 예외 없이 False를 반환.
  - start(): 실행 파일 존재를 확인하고, 시작 전 kill_leftover_processes()를
    다시 호출한 뒤 subprocess.Popen으로 백그라운드 기동한다.
  - is_running() / stop(): 프로세스 생존 확인, terminate() → (5초 내 미종료 시)
    kill()로 강제 종료. start() 없이 호출해도 예외 없음(방어적 설계).
  - capture_screenshot(path): output/<run_id>/assets/captured_sim.png 규격으로
    1024x600 PNG 저장. 화면 픽셀을 가져오는 부분은 _grab_frame() seam으로 분리해
    테스트에서 합성 프레임으로 대체 가능하게 설계.

2. 카드 12항(실패 시 대처) — 고스트 프로세스 강제 정리 검증
--------------------------------------------------------------------
검증 방법: tests/test_sim_driver.py::test_start_kills_leftover_ghost_process_before_launching_fresh_one

절차 (실제 MSYS2 툴체인 + 실제 build_sim/bin/lvgl_simulator.exe 대상):
  1. driver.build() 로 실제 실행 파일을 준비한다.
  2. 테스트 코드가 sim_driver 바깥에서 subprocess.Popen(executable_path)로
     "고스트" 프로세스를 직접 띄운다 (드라이버가 관리하지 않는 인스턴스).
  3. 1초 대기 후 ghost.poll() is None 으로 고스트가 살아있음을 확인한다.
  4. driver.start() 호출 → 내부적으로 kill_leftover_processes()가 먼저 실행되어
     taskkill /F /IM lvgl_simulator.exe 로 고스트를 강제 종료한 뒤, 새 프로세스를
     새로 기동한다.
  5. driver.process.pid != ghost.pid 로 "새 프로세스가 별개로 기동됐다"는 것과,
     is_running()이 True임을 확인한다.
  6. driver.stop() 으로 새 프로세스도 정리한다.

실행 결과 (pytest, 실제 실행 로그):

  → 실제 lvgl_simulator.exe 고스트 프로세스가 드라이버 자신의 start() 로직에
    의해 taskkill로 강제 종료된 뒤, 새 프로세스가 정상적으로 새로 기동됨을
    실측으로 확인했다(모킹 없음).

추가 확인: kill_leftover_processes()는 대상이 없을 때도(taskkill 실패/returncode!=0)
예외를 던지지 않고 False를 반환한다.

4. 실제 vs 대체(seam) 검증 범위 구분
--------------------------------------------------------------------
[실제 툴체인/실제 바이너리로 검증 — 모킹 없음]
  - build(): 실제 cmake configure + cmake --build 실행 → build_sim/bin/lvgl_simulator.exe
    산출 확인 (MSYS2 cmake 4.3.4 / gcc 16.1.0 / SDL2 2.32.10 사용).
  - start()/stop(): 실제 lvgl_simulator.exe 서브프로세스 기동 → is_running() True →
    5초 생존 보증 → stop()으로 정상 종료(terminate, 5초 내 반영) → is_running() False.
  - 카드 12항 고스트 정리: 실제 고스트 프로세스를 별도로 띄운 뒤, driver.start()
    자신의 로직이 taskkill /F /IM lvgl_simulator.exe 로 이를 강제 종료하고 새
    프로세스를 새로 기동하는 것을 PID 비교로 확인.

[테스트 seam(대체)으로 검증 — 화면 픽셀 캡처 부분만]
  - capture_screenshot()의 파일 저장/포맷(PNG 매직 넘버)/경로 규약
    (output/<run_id>/assets/captured_sim.png)/디렉토리 자동 생성/1024x600 크기는
    _grab_frame()을 합성(placeholder) 프레임으로 교체해 실제로 저장까지 수행,
    Pillow로 재오픈해 크기(1024, 600)와 PNG 매직 넘버를 검증했다.
  - _grab_frame()의 "실제" 구현은 PIL.ImageGrab.grab()으로 데스크톱 화면 영역을
    그랩하도록 되어 있으나, 이 자동화 세션(원격/헤드리스 셸 실행 환경)에서는
    SDL 창이 실제 데스크톱 전면에 표시되는지, 그리고 그 창 영역만 정확히
    크롭해 캡처할 수 있는지 육안/좌표 기준으로 확증할 수 없다. 따라서 이
    부분은 "테스트 가능한 seam으로 분리"하는 설계로 대응했고, 실제 라이브
    SDL 창 스크린샷의 화면 내용 자체를 이 세션에서 완전히 실증하지는
    못했음을 정직하게 명시한다. (T-801 검증기록의 "자동화 세션 육안 확인
    제한적" 한계와 동일한 성격의 제약이다.)

5. 환경 한계 명시
--------------------------------------------------------------------
- 이 자동화 세션은 원격 셸을 통해 명령을 실행하며, SDL 창이 물리적으로
  전면에 표시되는지 육안으로 확인할 수 없다(T-801과 동일한 제약).
  다만 프로세스가 SDL 관련 에러 출력 없이 5초 이상 정상 구동/종료되는 것은
  실측했다.
- capture_screenshot()의 _grab_frame() "실제" 경로(PIL.ImageGrab 기반 데스크톱
  캡처)는 이 세션에서 실행 가능함을 별도로 확인했으나(ImageGrab.grab() 호출
  성공, 해상도 2880x1800 반환), 이것이 "SDL 시뮬레이터 창 내용"인지 화면
  전체인지는 창 배치/포커스에 따라 달라질 수 있어 완전한 보증은 아니다.
  따라서 자동 테스트는 _grab_frame()을 합성 프레임으로 대체하는 seam을
  통해 저장 경로/포맷 규약만 확정적으로 검증했다.
- 이러한 한계에도 불구하고 카드 10항의 핵심 통과 기준(창이 팝업 기동 →
  5초 생존 → 에러 없이 자동/명시적 종료)과 카드 12항의 고스트 프로세스
  강제 정리는 실제 바이너리로 완전히 실증되었다.
