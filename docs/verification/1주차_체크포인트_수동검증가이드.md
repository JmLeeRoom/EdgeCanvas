## 2-A. [추가] 현재 사용 중인 PDF에 대한 안내와 실제 데이터시트로 재검증하는 방법

> 이 섹션은 "지금 Q1 검증이 진짜 ESP32-P4 데이터시트로 한 게 맞냐"는 질문에 답하기 위해 추가되었습니다. 기존 2-1~2-5 절차(합성 fixture 기준)는 그대로 유효하며, 아래는 **선택적 심화 재검증** 절차입니다.

### 2-A-1. 현재 상태: 합성 샘플 PDF를 사용했습니다

이번 T-008 검증(위 2-1~2-5, `docs/verification/T-008_extracted_tables.md`)은 **실제 Espressif ESP32-P4 공식 데이터시트가 아니라**, 저장소에 실제 데이터시트 PDF가 없어서 카드 8-2가 요구하는 "디스플레이 핀 구성 + 레지스터 맵"을 흉내낸 **합성(synthetic) 샘플 PDF** `tests/data/p4_datasheet_sample.pdf`(reportlab으로 생성, 5페이지, 5개 표)로 수행되었습니다. 이 사실은 `docs/verification/T-008_extracted_tables.md` 1장("입력 데이터 처리 방식")에 이미 기록되어 있습니다.

합성 PDF는 표 구조(핀맵/레지스터맵 컬럼 구성)만 실제 데이터시트를 흉내낸 것이라, Upstage Document Parse의 **표 추출 정확도 자체**는 검증할 수 있지만, 실제 ESP32-P4 데이터시트 특유의 복잡한 표(병합 셀, 다단 헤더, 각주, 90페이지 이상 분량)에 대한 강건성은 아직 실측하지 않았습니다.

### 2-A-2. 실제 데이터시트로 재검증하고 싶다면

**① 다운로드 링크 (2026-07-08 기준 확인, 아래 "유효성 확인 방법" 참조)**

- [ESP32-P4 Datasheet (Espressif 공식, Pre-release v0.6)](https://www.espressif.com/sites/default/files/documentation/esp32-p4_datasheet_en.pdf) — 93페이지, 약 1.43 MB(1,501,541 bytes), 핀맵/레지스터맵 등 표 다수 포함. **권장**: 표/레지스터맵 추출 검증에는 이 SoC 공식 데이터시트가 Waveshare 보드 문서보다 적합합니다(카드 7 목적이 "ESP32-P4 데이터시트"를 명시).
- [ESP32-P4 Technical Reference Manual (Espressif 공식)](https://www.espressif.com/sites/default/files/documentation/esp32-p4_technical_reference_manual_en.pdf) — 레지스터맵 표가 데이터시트보다 훨씬 방대함(참고용, 분량이 커서 1주차 검증 범위로는 과함).
- [ESP32-P4-WIFI6-Touch-LCD-7B 데이터시트 (Waveshare Wiki, 이 프로젝트의 실제 타깃 보드)](https://files.waveshare.com/wiki/common/Esp32-p4_datasheet_en.pdf) — Waveshare가 자사 서버에 미러링한 동일 Espressif 데이터시트(파일명만 다름). Waveshare 위키(`https://www.waveshare.com/wiki/ESP32-P4-WIFI6-Touch-LCD-7B`)의 "Datasheets" 절에서 확인.
  - 참고: `단위구현계획서.md`가 실제 타깃으로 삼는 보드는 **"Waveshare ESP32-P4" 개발보드**(T-004, T-501 등 카드 7/8에 명시)이지만, Waveshare는 보드 전용 데이터시트를 따로 두지 않고 **Espressif의 ESP32-P4 SoC 데이터시트/TRM을 그대로 링크**합니다. 즉 위 Espressif 링크와 Waveshare 링크는 **내용이 동일한 문서**이며, 표/레지스터맵 추출 검증 목적에는 Espressif 공식 URL을 기준으로 삼는 것이 더 안정적입니다(Waveshare 미러는 위키 개편 시 경로가 바뀔 수 있음).

**유효성 확인 방법과 결과**: `Invoke-WebRequest`로 실제로 HTTP 요청을 보내 확인했습니다. 두 링크 모두 `200 OK`, `Content-Type: application/pdf`로 정상 응답했습니다(추측 링크가 아니라 실제로 접속해 확인함). Espressif 링크는 다운로드해 페이지 수(93페이지)까지 실측했습니다. 검증 후 받은 파일은 저장소에 남기지 않고 즉시 삭제했습니다.

**② 저장할 경로**

프로젝트 관례(제6.2장, `tests/data/`)를 따라 아래 경로에 저장하세요(직접 다운로드해서 넣어야 합니다 — 이 문서는 링크 안내만 하며, 에이전트가 대신 다운로드/커밋하지 않습니다):

**주의 — 손실율(<5%) 기준 재적용 시 유의사항**:

1. `test_table_extraction`은 `EXPECTED_KEY_CELLS`(27~54행, `PIN_MAP_CELLS` + `REGISTER_MAP_CELLS`, 합성 PDF 전용 그라운드 트루스 셀 목록)를 기준으로 손실율을 계산합니다. 실제 데이터시트는 표 구성·셀 값이 완전히 다르므로, 이 그라운드 트루스 목록과 대조하면 거의 100% "손실"로 잘못 판정됩니다. 즉 **손실율 계산이 의미를 가지려면 `EXPECTED_KEY_CELLS`도 실제 데이터시트의 특정 표(예: Pin Overview 절의 핀맵 표)에서 직접 셀 값을 뽑아 새로 정의해야 합니다** — 이는 이 문서의 안내 범위를 넘는 별도 작업입니다.
2. 실제 데이터시트는 93페이지 전체를 파싱하면 표가 훨씬 많고(핀맵·레지스터맵 외 전력/타이밍 표 다수) API 처리 시간과 응답 크기가 합성 PDF(5페이지)보다 크게 늘어납니다. 필요하다면 관심 있는 표가 있는 페이지 구간만 잘라낸 부분 PDF를 만들어 테스트하는 것이 실용적입니다(전체 93페이지를 그대로 넣는 것은 스파이크 범위를 벗어남).

## 3. Q2 — Solar Pro 3 이미지 실험 (T-009)

**질문**: Solar Pro 3 fixture 이미지 3종 실험이 완료되었는가?

> **중요 — 결론 먼저 읽기**: 이 실험의 체크포인트 질문("실험이 **완료**되었는가")에 대한 답은 **YES(완료)** 입니다. 하지만 실험의 **기술적 결론은 No-Go** 입니다. 즉 Solar Pro 3(및 Solar Pro 2)는 채팅 엔드포인트에서 **이미지 입력을 지원하지 않고**, 실제 API가 `HTTP 400 "Image input is not allowed for this model"`로 명시적으로 거부합니다. 그래서 이미지 판정 메인 경로는 **OpenCV 픽셀 매칭 + 로컬 OCR fallback**으로 확정되었습니다. 당신이 눈으로 확인할 것은 이 두 가지 사실입니다: **(a) 이미지 입력이 실제로 거부되는 응답**, **(b) OpenCV+OCR fallback이 동작함**.

### 3-1. 실행 명령

```powershell
.venv\Scripts\python.exe -m pytest tests/test_solar_vision.py -v -s
```

- fixture 3종은 `tests/data/ui_normal.png`, `tests/data/ui_truncated.png`, `tests/data/ui_misplaced.png` 입니다(1024×600).
- 오프라인 테스트(인코딩·메시지 구성·OpenCV fallback)는 API 키 없이도 통과하고, 라이브 테스트는 `UPSTAGE_API_KEY`가 있을 때 실제 거부 응답을 관측합니다.

### 3-2. 눈으로 확인할 핵심 포인트

1. **이미지 입력 거부(가정 기각)**: `solar-pro3`와 `solar-pro2` 두 모델 모두 `HTTP=400` 과 `"Image input is not allowed for this model"` 이 출력됩니다.
2. **텍스트 전용은 정상**: 대조군 `test_text_only_chat_still_works`가 PASSED — API 키·엔드포인트 자체는 유효하다는 뜻.
3. **OpenCV fallback 동작**: `test_fallback_opencv_flags_misplaced_layout`(버튼 배치오류 → FAIL 검출)와 `test_fallback_opencv_passes_normal_layout`(정상 화면 → PASS) 둘 다 PASSED.
4. 맨 아래 `9 passed`.

## 4. Q3 — SDL2/CMake 시뮬레이터 빌드·창 (T-801)

**질문**: SDL2 + CMake 시뮬레이터 빌드가 1024×600 창을 띄우는가? (hello UI)

> Q3의 핵심은 **당신이 직접 창을 띄워 눈으로 보는 것**입니다. 이전 자동화 세션에서는 빌드·실행까지는 됐지만 "창이 데스크톱에 실제로 떠서 마우스에 반응하는지"를 육안으로 확인하지 못했습니다(`T-801_build_check.txt` 3장의 "육안 확인은 제한적" 참고). 이 단계에서 바로 그 육안 확인을 마무리합니다.

### 4-1. 빌드 도구 확인 (이미 설치돼 있을 수 있음)

Q3는 CMake·gcc·SDL2가 필요합니다. MSYS2로 설치돼 있다면 실행 파일은 `C:\msys64\mingw64\bin`에 있습니다. **현재 PowerShell 세션의 PATH에 이 경로를 먼저 추가**한 뒤 확인합니다.

```powershell
$env:Path = "C:\msys64\mingw64\bin;" + $env:Path
cmake --version
gcc --version
```

기대 출력(버전 숫자는 다를 수 있음):

```
cmake version 4.3.4
gcc.exe (Rev5, Built by MSYS2 project) 16.1.0
```

- 위 두 명령이 버전을 출력하면 도구가 준비된 것입니다. 다음 단계로 갑니다.
- `'cmake' ... 인식되지 않습니다` 같은 오류가 나면 아직 설치가 안 됐거나 경로가 다른 것입니다 → **4-2**로 가서 설치하세요.

> 참고: `sdl2-config`는 셸 스크립트라 PowerShell에서 직접 `sdl2-config --version`을 실행하면 오류처럼 보일 수 있습니다. SDL2 자체는 CMake의 `find_package(SDL2)`가 찾으므로, `cmake`/`gcc`만 인식되면 다음 단계로 진행해도 됩니다.

### 4-2. (도구가 없을 때만) MSYS2 + SDL2 설치

이미 도구가 확인됐다면 이 절은 건너뜁니다.

```powershell
winget install --id MSYS2.MSYS2
C:\msys64\usr\bin\bash.exe -lc "pacman -Sy --noconfirm --needed mingw-w64-x86_64-toolchain mingw-w64-x86_64-cmake mingw-w64-x86_64-SDL2"
```

설치 후 다시 **4-1**의 PATH 추가와 `cmake --version` / `gcc --version` 확인을 수행합니다.

### 4-3. 빌드

같은 PowerShell 세션(4-1에서 PATH를 추가한 그 창)에서 실행합니다.

```powershell
cmake -S src/simulator -B build_sim -G "MinGW Makefiles" -DCMAKE_PREFIX_PATH="C:/msys64/mingw64"
cmake --build build_sim
```

- 첫 번째 명령은 SDL2를 찾고 LVGL(release/v9.2)을 FetchContent로 내려받아 빌드를 구성합니다(`Configuring done / Generating done`).
- 두 번째 명령이 끝나면 `build_sim\bin\lvgl_simulator.exe`(약 1.2 MB)가 생성됩니다.

빌드 성공을 스크립트로 판정할 수도 있습니다(선택):

```powershell
powershell -File src\simulator\check_build.ps1
```

`PASS: lvgl_simulator 실행 파일이 생성되었습니다.` 및 종료 코드 0이면 성공입니다(0=성공, 1=빌드 실패, 2=도구 부재).

### 4-4. 창을 직접 띄워 눈으로 확인 (Q3의 핵심)

```powershell
& .\build_sim\bin\lvgl_simulator.exe
```

또는 파일 탐색기에서 `d:\EdgeCanvas\build_sim\bin\lvgl_simulator.exe`를 **더블클릭**해도 됩니다.

> **SDL2.dll 안내**: exe 는 `SDL2.dll` 을 동적 링크하므로 실행 시 이 DLL 을 찾아야 합니다. `CMakeLists.txt` 의 빌드 후 자동 복사(POST_BUILD)로 `build_sim\bin\SDL2.dll` 이 exe 옆에 함께 생성되므로 **더블클릭만으로 바로 실행**됩니다. 만약 `SDL2.dll이 없어 코드 실행을 진행할 수 없습니다` 오류 창이 뜨면 아래 **6. 문제 발생 시 → Q3** 표를 참고하세요.

**눈으로 확인할 핵심 포인트**:

1. **1024×600 크기의 창**이 화면에 뜬다.
2. 창 안 가운데에 흰 글씨로 **hello 라벨**이 보인다:
   `P10 Manufacturing` / `LVGL 9.x SDL2 Simulator` / `1024 x 600` (배경은 짙은 회색 `#1e1e1e`).
3. 창 위에서 **마우스를 움직이거나 클릭**했을 때 프로그램이 멈추지 않고 정상 반응한다(입력 드라이버 `lv_sdl_mouse_create()` 동작).

> 이 프로그램은 무한 루프로 창을 계속 띄웁니다. 확인이 끝나면 창의 X(닫기) 버튼을 누르거나, 명령으로 실행한 경우 PowerShell에서 `Ctrl+C`로 종료하세요.

참고로, 이전 세션에서 저장한 구동 스냅샷도 함께 볼 수 있습니다.

```powershell
start docs\verification\T-801_pc_simulator_run.png
```

### Q3 (T-801)

| 증상 | 원인 | 대처 |
|------|------|------|
